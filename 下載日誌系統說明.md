# 下載日誌系統說明

## 📋 功能概述

每次下載都會自動記錄詳細的日誌信息，包括：
- **檔案屬性**：URL、大小、類型、MIME 類型
- **來源追蹤**：從哪個函數觸發的下載
- **命名原則**：如何生成的檔名
- **解析位置**：從哪裡解析到的圖片
- **下載狀態**：排隊、完成、失敗、中斷

## 📊 日誌記錄內容

### 1. 檔案屬性 (fileInfo)
```javascript
{
  url: "完整的圖片 URL",
  urlKey: "URL 的前 200 字符（用於去重）",
  urlLength: 1234, // URL 長度
  fileSize: 2048576, // 文件大小（字節）
  fileSizeFormatted: "2.0 MB", // 格式化的大小
  mimeType: "image/png", // MIME 類型
  imageType: "highres" // 'thumbnail' 或 'highres'
}
```

### 2. 來源信息 (source)
```javascript
{
  function: "forceExtractRealImage", // 觸發下載的函數名
  chatId: "abc123...", // 對話 ID
  conversationTitle: "對話標題", // 對話標題
  isManual: false // 是否為手動下載
}
```

### 3. 命名原則 (naming)
```javascript
{
  rule: "provided" | "generated", // 檔名是提供的還是生成的
  originalFilename: "原始檔名.png", // 原始檔名
  finalFilename: "最終檔名_1234567890_abc.png", // 最終檔名
  downloadPath: "Gemini_Assistant/對話標題/檔名.png", // 完整下載路徑
  details: {
    // 命名規則的詳細信息
  }
}
```

### 4. 下載狀態 (download)
```javascript
{
  downloadId: 12345, // Chrome 下載 ID
  status: "queued" | "completed" | "failed" | "interrupted", // 下載狀態
  error: "錯誤信息（如果有）"
}
```

### 5. 識別信息 (identifiers)
```javascript
{
  requestId: "穩定的 requestId",
  originalRequestId: "原始的 requestId",
  stableRequestId: "穩定的 requestId（用於去重）"
}
```

## 🔍 來源函數標識

系統會自動識別以下來源函數：

1. **`forceExtractRealImage`** - 強制提取定時器（每 5 秒）
2. **`extractGeneratedImages`** - 圖片提取函數
3. **`extractGeneratedImages_loaded`** - 圖片提取（loaded 狀態）
4. **`observeImageLoading`** - 圖片載入監聽
5. **`observeImageLoading_immediate`** - 圖片載入監聽（立即下載）
6. **`observeImageLoading_srcChange`** - 圖片載入監聽（src 變化）
7. **`checkImageSizeAndDownload_highres`** - 圖片分級（高畫質）
8. **`checkImageSizeAndDownload_thumbnail`** - 圖片分級（預覽圖）
9. **`setupImageObserver`** - 圖片觀察器設置
10. **`getRealImagePath`** - 獲取真實圖片路徑
11. **`clickDownloadButtonByIndex`** - 手動點擊下載按鈕
12. **`extractImages`** - 提取圖片（從消息中）
13. **`scrapeMessages`** - 抓取消息

## 📝 日誌查看方式

### 1. 控制台日誌
每次下載時，控制台會輸出詳細信息：
```
[Background] [下載日誌] ========== 下載記錄 ==========
[Background] [下載日誌] 來源函數: forceExtractRealImage
[Background] [下載日誌] URL 長度: 1234
[Background] [下載日誌] 檔案大小: 2.0 MB
[Background] [下載日誌] 命名規則: provided
[Background] [下載日誌] 原始檔名: image.png
[Background] [下載日誌] 最終檔名: image_1234567890_abc.png
[Background] [下載日誌] 下載路徑: Gemini_Assistant/對話標題/image_1234567890_abc.png
[Background] [下載日誌] 下載狀態: completed
[Background] [下載日誌] RequestId: img_abc123...
[Background] [下載日誌] ==============================
```

### 2. 操作日誌系統
所有下載日誌都會保存到 `operation_logs_default` 中，可以通過 Side Panel 的「操作日誌」功能查看和導出。

### 3. 導出日誌
在 Side Panel 中點擊「導出操作日誌」，可以導出所有下載記錄為 TXT 文件。

## 🎯 使用場景

### 場景 1：分析重複下載
通過查看日誌，可以找出：
- 哪些函數觸發了重複下載
- 同一個 URL 被下載了多少次
- 每次下載的來源是什麼

### 場景 2：取消特定監聽
根據日誌中的 `source.function`，可以：
- 找出重複下載最多的函數
- 逐步取消或優化該函數的監聽邏輯

### 場景 3：分析命名規則
通過查看 `naming` 信息，可以：
- 了解檔名是如何生成的
- 找出命名規則的問題
- 優化命名邏輯

## 🔧 後續優化建議

1. **根據日誌分析**：導出日誌後，分析哪些函數觸發下載最多
2. **逐步取消監聽**：從觸發次數最多的函數開始，逐步取消或優化
3. **驗證效果**：每次取消後，重新導出日誌，驗證重複下載是否減少

## 📌 注意事項

1. 日誌會自動清理，只保留最近 1000 條記錄
2. 如果存儲空間不足，會自動清理舊日誌
3. 日誌包含完整的 URL 信息，注意隱私保護
