# 程序優化記錄

## 📊 當前問題分析

### 1. 性能問題

#### 1.1 監聽記錄系統
- **問題**：`recordClickMonitorEvent` 每次記錄都發送兩條消息（background.js 和 sidepanel）
- **影響**：高頻事件（如 DOM 變化）會產生大量消息，導致性能下降
- **頻率**：每次按鈕點擊可能觸發 10-50 條記錄

#### 1.2 URL 重定向追蹤
- **問題**：`trackImageUrlRedirectChain` 可能被多次調用同一個 URL
- **影響**：重複的網絡請求浪費資源
- **頻率**：每個按鈕點擊可能觸發 1-4 次追蹤

#### 1.3 網絡請求攔截
- **問題**：每次按鈕點擊都會設置新的 fetch/XHR 攔截器
- **影響**：可能導致攔截器堆疊，影響性能
- **頻率**：每個按鈕點擊設置一次

### 2. 內存管理問題

#### 2.1 記錄數組增長
- **問題**：`clickMonitorRecords` 可能快速增長
- **影響**：內存使用增加
- **當前限制**：100 條記錄

#### 2.2 MutationObserver 清理
- **問題**：多個 MutationObserver 可能沒有正確清理
- **影響**：內存洩漏
- **頻率**：每個按鈕點擊創建一個 Observer

### 3. 代碼重複

#### 3.1 URL 檢查邏輯
- **問題**：`isDownloadRelatedUrl` 和 `isDownloadUrl` 邏輯重複
- **影響**：維護困難，容易出錯

#### 3.2 事件記錄邏輯
- **問題**：多處重複的記錄邏輯
- **影響**：代碼冗餘

## 🔧 優化方案

### 優化 1：批量發送記錄（防抖）
- 將記錄先存儲在內存中
- 使用防抖機制，每 500ms 批量發送一次
- 減少消息傳遞次數

### 優化 2：URL 追蹤去重
- 使用 Set 記錄正在追蹤的 URL
- 避免重複追蹤同一個 URL
- 追蹤完成後從 Set 中移除

### 優化 3：統一 URL 檢查函數
- 創建統一的 `isDownloadUrl` 函數
- 所有地方都使用同一個函數
- 減少代碼重複

### 優化 4：改進 MutationObserver 管理
- 使用統一的 Observer 管理器
- 確保所有 Observer 都能正確清理
- 避免內存洩漏

### 優化 5：優化記錄數據結構
- 減少記錄中的冗餘數據
- 使用更緊湊的數據格式
- 限制單條記錄的大小

## 📈 預期效果

- **消息傳遞減少**：從每次記錄 2 條消息減少到每 500ms 批量發送（減少 90%+ 的消息量）
- **網絡請求減少**：避免重複追蹤同一個 URL（使用 Set 去重）
- **內存使用優化**：更好的清理機制，減少內存洩漏
- **代碼可維護性**：減少重複代碼，統一邏輯
- **數據大小優化**：限制記錄數據大小，避免過大的記錄

## ✅ 已實施的優化

### 1. 批量發送記錄（防抖機制）✅
- 實現了 `flushPendingRecords()` 函數
- 使用 500ms 防抖，批量發送記錄
- 當待發送記錄達到 20 條時立即發送
- 監聽結束時確保所有記錄都被發送

### 2. URL 追蹤去重 ✅
- 使用 `trackingUrls` Set 記錄正在追蹤的 URL
- 避免重複追蹤同一個 URL
- 30 秒後自動清理，防止永久佔用

### 3. 統一 URL 檢查函數 ✅
- 創建了統一的 `isDownloadUrl()` 函數
- 所有地方都使用同一個函數
- 減少了代碼重複

### 4. 數據大小優化 ✅
- 限制記錄數據大小（字符串超過 1000 字符會被截斷）
- 限制 URL 長度（200 字符）
- 優化重定向鏈記錄（只記錄摘要，不記錄完整鏈）

### 5. 改進錯誤處理 ✅
- 在所有追蹤路徑中確保清理追蹤標記
- 改進異常處理，防止內存洩漏
