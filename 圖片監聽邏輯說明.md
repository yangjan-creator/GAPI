# 圖片監聽解析下載的頻率和邏輯說明

## 📊 監聽頻率總覽

### 1. **強制提取定時器 (forceExtractInterval)**
- **頻率**: 每 **5 秒**掃描一次
- **函數**: `forceExtractRealImage()`
- **觸發條件**: 只要 runtime 有效就執行（不檢查對話 ID）
- **掃描範圍**: 
  - `button.image-button` 內的圖片
  - `single-image` 內的圖片
  - 所有包含 `googleusercontent.com` 的圖片

### 2. **圖片檢查定時器 (imageCheckInterval)**
- **頻率**: 每 **3 秒**檢查一次
- **函數**: 檢查圖片數量變化，觸發 `extractGeneratedImages()`
- **觸發條件**: 圖片數量變化時觸發

### 3. **MutationObserver 監聽器**
- **imageObserver**: 監聽整個 `document.body` 的 DOM 變化
- **setupImageObserver**: 為每個圖片容器設置獨立的 MutationObserver
- **observeImageLoading**: 監聽單個圖片的 `src` 屬性變化

### 4. **對話提取時的圖片掃描**
- 在 `scrapeMessages()` 中，每次提取消息時會調用 `extractImages()`
- 如果沒有找到圖片，會在 **1 秒、3 秒、5 秒**後進行二次掃描（共 3 次）

## 🔄 為什麼一張圖會觸發 30 次？

### 問題根源分析

#### 1. **多個監聽器同時工作**
```
同一張圖片可能被以下多個機制同時捕獲：
├─ forceExtractRealImage() (每 5 秒)
├─ extractGeneratedImages() (每 3 秒，當圖片數量變化時)
├─ imageObserver (DOM 變化時)
├─ setupImageObserver() (為每個容器設置)
├─ observeImageLoading() (監聽 src 變化)
└─ extractImages() (在 scrapeMessages 中，1秒/3秒/5秒重試)
```

#### 2. **requestId 每次都不同**
在 `triggerAutoDownload()` 函數中（第 5105 行）：
```javascript
requestId: requestId + '_' + Date.now(), // 確保 requestId 永遠不重複
```
這導致即使同一張圖片，每次調用時 `requestId` 都不同，去重檢查失效。

#### 3. **去重檢查的時序問題**
```javascript
// 在 triggerAutoDownload 中
const checkResult = await checkDownloadHistory(imageUrl, requestId, currentChatId);
if (checkResult.exists) {
  return; // 應該跳過
}
```
但由於：
- `requestId` 每次都不同（加了 `Date.now()`）
- 多個監聽器幾乎同時觸發
- Storage 讀取是異步的，可能還沒寫入就再次檢查

#### 4. **processedImageUrls 只在內存中**
```javascript
let processedImageUrls = new Set(); // 僅在記憶體中
```
這個 Set 在以下情況會被清空：
- 頁面刷新
- 切換對話（但可能還沒執行清理）
- 手動下載時（`processedImageUrls.clear()`）

#### 5. **多個掃描路徑重複處理**
同一張圖片可能被以下路徑重複處理：
- `extractGeneratedImages()` → 遍歷所有 `button.image-button`
- `extractGeneratedImages()` → 遍歷所有 `img` 標籤
- `forceExtractRealImage()` → 再次遍歷所有 `button.image-button`
- `setupImageObserver()` → 為每個容器設置監聽器
- `observeImageLoading()` → 監聽 src 變化

## 📈 觸發頻率計算

假設一張圖片在頁面上存在 **30 秒**：

1. **forceExtractRealImage**: 30秒 ÷ 5秒 = **6 次**
2. **imageCheckInterval**: 30秒 ÷ 3秒 = **10 次**（如果圖片數量變化）
3. **extractGeneratedImages**: 在對話提取時觸發，可能 **3-5 次**
4. **MutationObserver**: DOM 變化時觸發，可能 **5-10 次**
5. **二次掃描重試**: 1秒、3秒、5秒各一次 = **3 次**

**總計**: 6 + 10 + 5 + 10 + 3 = **34 次**（接近 30 次）

## 🔧 優化建議

### 1. **統一 requestId 生成邏輯**
```javascript
// 問題：每次都加 Date.now()，導致 requestId 不同
requestId: requestId + '_' + Date.now()

// 建議：使用 URL 的 hash 作為 requestId
const urlHash = btoa(imageUrl.substring(0, 200)).substring(0, 20);
requestId: urlHash || requestId
```

### 2. **加強去重檢查**
```javascript
// 在 triggerAutoDownload 開頭立即檢查
const urlKey = getUrlKey(imageUrl);
if (processedImageUrls.has(urlKey)) {
  console.log('⏭️ 跳過：已在內存中處理過');
  return;
}

// 先標記為已處理（防止並發）
processedImageUrls.add(urlKey);

// 然後檢查持久化資料庫
const checkResult = await checkDownloadHistory(imageUrl, requestId, currentChatId);
if (checkResult.exists) {
  console.log('⏭️ 跳過：已在資料庫中');
  return;
}
```

### 3. **減少掃描頻率**
- `forceExtractInterval`: 從 5 秒改為 **10 秒**
- `imageCheckInterval`: 從 3 秒改為 **5 秒**
- 二次掃描重試: 只在第一次掃描失敗時才重試

### 4. **統一圖片處理入口**
創建一個統一的圖片處理函數，所有監聽器都調用它：
```javascript
async function processImageOnce(imageUrl, requestId, source) {
  const urlKey = getUrlKey(imageUrl);
  
  // 立即檢查內存緩存
  if (processedImageUrls.has(urlKey)) {
    return { processed: false, reason: 'memory_cache' };
  }
  
  // 標記為處理中
  processedImageUrls.add(urlKey);
  
  // 檢查持久化資料庫
  const checkResult = await checkDownloadHistory(imageUrl, requestId, currentChatId);
  if (checkResult.exists) {
    return { processed: false, reason: 'database' };
  }
  
  // 執行下載
  await triggerAutoDownload(imageUrl, requestId);
  return { processed: true };
}
```

### 5. **優化 MutationObserver**
- 一旦圖片被處理，立即停止對該節點的觀察
- 使用 `observerManager` 統一管理，避免重複觀察

## ✅ 已實施的解決方案

### 修復內容

#### 1. **使用穩定的 requestId**
- **修復前**: `requestId: requestId + '_' + Date.now()` （每次都不同）
- **修復後**: 使用 URL 的 hash 作為穩定的 requestId
```javascript
const stableRequestId = urlKey.substring(0, 50) + '_' + btoa(urlKey).substring(0, 10);
```

#### 2. **加強內存緩存檢查**
- 在 `triggerAutoDownload` 開頭立即檢查 `processedImageUrls`
- 立即標記為已處理，防止並發調用
- 下載失敗時從緩存中移除，允許重試

#### 3. **優化去重檢查邏輯**
- 優先使用 URL 的 hash 進行檢查（穩定）
- 兼容舊數據（也檢查 requestId）
- 使用 `getUrlKey()` 統一提取 URL 關鍵部分

#### 4. **改進資料庫記錄**
- 使用 URL 的 hash 作為主鍵
- 同時保存完整 URL 和 requestId（用於兼容和調試）

### 預期效果

修復後，同一張圖片應該：
- **最多觸發 1 次下載**（而不是 30 次）
- **內存緩存立即生效**，防止並發
- **持久化資料庫檢查準確**，使用穩定的 URL hash

### 剩餘優化建議

1. **減少掃描頻率**
   - `forceExtractInterval`: 從 5 秒改為 10 秒
   - `imageCheckInterval`: 從 3 秒改為 5 秒

2. **統一圖片處理入口**
   - 創建 `processImageOnce()` 函數
   - 所有監聽器都通過它處理圖片

3. **優化 MutationObserver**
   - 圖片處理後立即停止觀察
   - 使用 `observerManager` 統一管理
